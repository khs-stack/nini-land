# DECISIONS.md — 주요 설계 결정 기록

날짜 형식과 사유를 남겨 추후 "왜 이렇게 만들었는지"를 추적한다.

## 2026-07-18 — 프로젝트 시작 방식
- 기존 저장소/코드 없음을 확인. 새 프로젝트를 생성하는 대신 기획 문서(docs/)부터 작성.
- 이유: 지시서 3항 "코드를 바로 수정하지 말고 문서를 먼저 읽어라" 및 17항 "기획문서 작성이
  완료되면 코드를 수정하지 말고 멈춰라"에 따름.

## 2026-07-18 — 도매가 노출 방식
- 도매가는 CSS/조건부 렌더링으로 숨기지 않고, 서버에서 승인된 도매회원 여부를 확인한 뒤에만
  API 응답에 필드 자체를 포함시키는 방식으로 결정.
- 이유: 브라우저 개발자도구/페이지소스/API 응답에서 비인가 사용자가 도매가를 확인할 수 없어야
  한다는 보안 요구사항(지시서 6항, 15항)을 코드 레벨에서 원천 차단하기 위함.

## 2026-07-18 — 결제 모듈 분리
- PG사 미확정 상태이므로 결제 인터페이스를 추상화하고 테스트 결제 환경으로 우선 구현.
- 이유: "특정 PG사에 코드가 지나치게 종속되지 않도록" 요구사항 반영. 추후 PG 계약 시 어댑터만
  교체 가능하도록.

## 2026-07-18 — 네이버 연동은 설정값만 우선 준비
- 네이버 플레이스/스마트스토어/톡톡/페이지 연동은 이번 단계에서 구현하지 않고, 관리자 설정
  테이블(`external_service_settings`)과 모듈 자리만 마련.
- 이유: 자체 주문 기능이 핵심이며 네이버 이동 방식으로 대체하지 않는다는 원칙(지시서 10항)을
  지키면서도 추후 확장 시 코드 구조 변경을 최소화하기 위함.
